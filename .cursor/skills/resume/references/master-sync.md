# Reference: Keep master.md canonical

`master.md` is the sole source of truth for resume facts. Tailored `resume.md` files derive from it — never the other way around.

## User corrections and additions

When the user states experience, scope, technology, or framing that is **missing or incomplete** in `master.md`:

1. **Treat the user's statement as correct** — do not push back, hedge, or draft around the gap
2. **Update `master.md` first** with the new or corrected facts
3. **Then** draft or update the role's `resume.md`

This applies during interview answers, review feedback, and casual clarifications — not only when the user explicitly asks to edit master.

## What to add

- New projects, products, or responsibilities
- Accurate scope (e.g. sole vs co-built, from scratch vs contributed)
- Technology stack details (cloud, IaC, frameworks, architecture)
- Terminology the user prefers (e.g. product vs framework vs pipeline)

## What not to do

- Assume the user misspoke because master lacks support
- Leave gaps in master and omit claims from the tailored CV
- Fabricate beyond what the user (or master) supports

## After updating master

- Record significant additions in the role's `notes.md` under `## Master updates`
- If a general CV exists and should reflect master, sync it when the user asks or when changes are broad
